Environments

About Environments

Whatever the environment is, there is probably some level of a Development for code building, QA or Test for code validation, Stage for testing production data before release, and Production facing the user base. As noted in server settings, the system does have an environment variable setting for 'dev', 'qa', 'stage' and 'prod' for supporting the ease of moving project data from one environment to another. If there are already DNS FQDN resolving to different environments, you can pre-configure those resolutions for the project. Proxy Mapping and DNS FQDN resolution are attached to the project configuration and move with the project data. The servers in each environment will map according to what it's set for so there is no further configration needed when promoting application code to the next environment.

Use Example With APIs

The environment variable is also passed through as a server variable to server side execution code. It can be used where you have a config file that has different database in each environment as an example. Your code can be written to take the environment variable and get the configration for that environment type. An example of the server side properties that are passed to a server are as follows:

Request example using default API file in the Project Files management:
